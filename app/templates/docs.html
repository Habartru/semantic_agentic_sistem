Документация пайплайна SEO Agents

Ниже описано полное внутреннее устройство системы: как данные проходят через 8 агентов, какие внешние сервисы задействованы, где используется LLM, а где — только локальная обработка. Все описания основаны на реальном исходном коде.

Общая схема пайплайна

Пользовательский ввод (URL, seed-ключи, конкуренты, описание бизнеса, гео) ↓ ResearchAgent — парсинг sitemap, мета-данных конкурентов (локально) ↓ ExpansionAgent — Google Suggest + Google Trends + LLM (2 итерации) ↓ CleaningAgent — дедупликация + LLM-фильтрация ↓ IntentAgent — LLM-классификация по 7 типам интента ↓ ClusteringAgent — LLM-кластеризация (fallback по интенту) ↓ MappingAgent — LLM-привязка к страницам (fallback транслитерация) ↓ PrioritizationAgent — LLM-оценка по 7 факторам + взвешенная формула ↓ FeedbackAgent — заглушка для мониторинга ↓ Сохранение в БД (KeywordResult) + SSE-прогресс
Агентов в цепочке
8
Внешних сервисов
4
LLM-вызовов
6 методов
База данных
SQLite async

Доступные LLM-модели через OpenRouter

Все LLM-вызовы идут через OpenRouter API (совместимый с OpenAI SDK). Модель по умолчанию задаётся в переменной окружения OPENROUTER_MODEL или на странице настроек. Рекомендуемые модели: Qwen 3.6 Plus (по умолчанию), Kimi K2.6, GLM 5.1.

base_url = "https://openrouter.ai/api/v1" model = settings.OPENROUTER_MODEL # fallback: "qwen/qwen3.6-plus" temperature варьируется от 0.2 до 0.7 в зависимости от задачи

Для каждого вызова используется retry-логика: 3 попытки с экспоненциальным backoff (2, 4, 8 сек). Если LLM вернул невалидный JSON — делается до 2 дополнительных попыток с уточняющим промптом.

1
ResearchAgent — Исследование сайта и конкурентов
Анализирует сайт клиента и конкурентов, извлекает существующие страницы и мета-данные
Локальная обработка Внешний API
Входные данные
  • site_url — URL сайта клиента (опционально)
  • seed_keywords — стартовые ключевые слова из формы проекта
  • competitor_urls — список URL конкурентов
  • business_description — описание бизнеса
  • geo — гео-таргетинг (по умолчанию "Москва")
Внешние сервисы

CompetitorService (httpx + BeautifulSoup + lxml). User-Agent: Mozilla/5.0 (compatible; SEOBot/1.0). Для каждого конкурента:

  • Поиск sitemap по путям: /sitemap.xml, /sitemap_index.xml, а также через директиву Sitemap: в robots.txt
  • Парсинг XML через lxml, извлечение всех <loc>
  • Фильтрация URL: исключаются глубокие страницы товаров/статей (регулярные выражения), остаются категории и короткие пути (1-3 сегмента)
  • Для до 20 страниц на конкурента парсится HTML: title, meta description, h1, первые 5 h2
  • Задержка между запросами к страницам: 1-2 сек
Локальная обработка
  • _extract_keywords_from_pages — извлекает ключевые слова из title и h1 конкурентов. title и h1 добавляются как потенциальные ключи, первая часть title (до 6 слов) также извлекается после удаления разделителей (|, -, —, :)
  • _build_business_context — формирует строку контекста из описания бизнеса, гео и топ-20 категорий конкурентов
  • Объединение seed-ключей с извлечёнными, дедупликация через dict.fromkeys
LLM обработка

Не используется. Весь агент работает локально.

Выходные данные
{ "existing_pages": ["https://client.ru/page1", ...], "competitor_pages": [{"url": "...", "title": "...", "h1": "...", "description": "..."}], "competitor_keywords": ["категория 1", "категория 2"], "seed_keywords": ["ключ 1", "ключ 2", ...], "business_context": "Описание бизнеса: ...\nГео-таргетинг: ...\nКатегории конкурентов: ..." }
2
ExpansionAgent — Расширение семантического ядра
Итеративный цикл расширения ключевых слов через Google Suggest, Google Trends и LLM
Google Suggest Google Trends LLM
Входные данные
  • seed_keywords — объединённые seed + ключи из ResearchAgent
  • business_context — контекст бизнеса
  • geo — гео-таргетинг
Внешние сервисы

Google Suggest API — endpoint suggestqueries.google.com/complete/search?client=firefox. Параметры: hl=ru, gl=ru. Ответ парсится как JSON-массив, извлекается второй элемент (список подсказок). Между запросами задержка 0.5-1.5 сек. При ошибке — 3 попытки.

Google Trends — библиотека pytrends (TrendReq(hl="ru", tz=180)). Метод related_queries() возвращает top и rising запросы. Ограничение pytrends — 5 ключей за раз, поэтому разбиваем на батчи. Между батчами пауза 5 сек, при rate limit (429) — пауза 60 сек.

Локальная обработка
  • Объединение результатов из трёх источников в множество all_candidates
  • Источники каждого кандидата сохраняются в sources: google_suggest_iter1/2, google_trends_iter1, llm_expand_iter1/2
  • Финальная дедупликация через сортировку множества
LLM обработка

Метод LLMService.expand_keywords. Промпт запрашивает генерацию long-tail фраз на основе seed-ключей и бизнес-контекста. Используются 8 типов модификаторов:

  • Коммерческие: купить, цена, стоимость, заказать
  • Гео-модификаторы: город, район, метро
  • Характеристики: цвет, размер, материал, бренд
  • Назначение: для чего, кому, в каких условиях
  • Аудитория: для новичков, профессионалов, детей
  • Проблемы: сломалось, не работает, нужно исправить
  • Сравнения: лучше, дешевле, аналоги, отличия
  • Размеры/материалы: конкретные параметры

Temperature: 0.7. Формат ответа: JSON-массив строк. Исходные seed-ключи исключаются из результата.

Алгоритм итераций
Итерация 1 (depth=2 для Suggest): - Google Suggest для всех seed-ключей (depth=2: seed -> top-5 подсказок -> их подсказки) - Google Trends related queries для seed-ключей - LLM expand_keywords для seed-ключей Итерация 2 (только если найдены новые ключи): - Берём top-10 новых ключей из итерации 1 - Google Suggest для top-10 (depth=1) - LLM expand_keywords для top-10
Выходные данные
{ "candidates": ["фраза 1", "фраза 2", ...], "sources": { "фраза 1": ["google_suggest_iter1"], "фраза 2": ["llm_expand_iter1", "google_suggest_iter2"] } }
3
CleaningAgent — Очистка ключевых слов
Удаляет дубли, мусор и нерелевантные запросы через программную фильтрацию и LLM
Локальная обработка LLM
Входные данные
  • candidates — полный список кандидатов от ExpansionAgent
  • business_context — контекст для проверки релевантности
Локальная обработка
  • Дедупликация (_deduplicate): lowercase + strip, сохранение порядка через set
  • Фильтрация по длине: удаляются запросы короче 3 символов
LLM обработка

Метод LLMService.clean_keywords. Обработка батчами по 50 ключей. Temperature: 0.2.

Критерии удаления (keep = false):

  • Семантические дубли — идентичные или почти идентичные фразы
  • Мусор — бессмысленные сочетания слов без поискового интента
  • Нерелевантные — не связаны с бизнесом, продуктом или услугой
  • Юридически рискованные — контрафакт, запрещённые товары, медицинские утверждения без лицензии, гарантии результата
  • Слишком общие — одно слово без контекста
  • Нецелевые — относятся к совершенно другой нише

Формат ответа: JSON-массив [{"keyword": "...", "keep": true/false, "reason": "..."}]. При ошибке LLM возвращаются все ключи без фильтрации (graceful degradation).

Выходные данные
{ "cleaned_keywords": ["фраза 1", "фраза 2", ...], "removed": [ {"keyword": "мусор", "reason": "Удалено по решению LLM"}, {"keyword": "x", "reason": "Слишком короткий запрос (< 3 символов)"} ], "stats": { "total": 500, "kept": 420, "removed": 80, "deduped": 10, "too_short": 5, "llm_rejected": 65 } }
4
IntentAgent — Определение поискового интента
Классифицирует каждый запрос по 7 типам интента с confidence score
LLM
Входные данные
  • cleaned_keywords — очищенный список
  • business_context — контекст бизнеса
LLM обработка

Метод LLMService.classify_intent. Батчинг по 30 ключей. Temperature: 0.3.

7 типов интента:

  • commercial — пользователь готов к покупке, ищет товар/услугу (купить, цена, заказать)
  • transactional — сильная покупательская мотивация, готов совершить сделку прямо сейчас
  • informational — ищет информацию, ответ, инструкцию (как, что такое, зачем)
  • local — гео-зависимый запрос, связанный с конкретным городом/районом
  • comparison — сравнивает варианты, ищет лучший выбор (vs, лучше, отличия)
  • navigational — ищет конкретный сайт или бренд
  • problem_based — описывает проблему/боль, ищет решение (сломалось, болит, не работает)

Также определяется рекомендуемый тип страницы: category, product, article, landing, comparison, faq, home.

Формат ответа: [{"keyword": "...", "intent": "...", "confidence": 0.95, "page_type": "..."}]. Confidence — число от 0 до 1. При ошибке LLM все ключи получают intent: "unknown" и confidence: 0.0.

Выходные данные
{ "keywords_with_intents": [ {"keyword": "купить ноутбук", "intent": "commercial", "confidence": 0.92, "page_type": "category"}, {"keyword": "как выбрать ноутбук", "intent": "informational", "confidence": 0.88, "page_type": "article"} ] }
5
ClusteringAgent — Кластеризация по интенту
Группирует ключевые слова так, чтобы каждый кластер отвечал одному интенту = одной странице
LLM Fallback
Входные данные
  • keywords_with_intents — список словарей с ключами, интентами и confidence
LLM обработка

Метод LLMService.cluster_keywords. Все ключи передаются одним запросом. Temperature: 0.3.

Правила кластеризации в промпте:

  • Один кластер = один поисковый интент = одна страница сайта
  • Семантически похожие фразы с разным интентом — в разные кластеры (пример: "купить ноутбук" commercial vs "как выбрать ноутбук" informational)
  • Не создавать слишком мелких кластеров (менее 2 фраз) при явном совпадении интента
  • Не объединять фразы с принципиально разным интентом

Формат ответа: [{"cluster_name": "...", "main_keyword": "...", "keywords": [...], "intent": "...", "recommended_page_type": "..."}].

Fallback (локальная обработка)

Если LLM недоступен — простая группировка по полю intent через collections.defaultdict. Для каждого интента создаётся кластер с названием "Кластер {intent}".

Выходные данные
{ "clusters": [ { "cluster_name": "Покупка ноутбуков", "main_keyword": "купить ноутбук", "keywords": ["купить ноутбук", "ноутбук цена", "заказать ноутбук"], "intent": "commercial", "recommended_page_type": "category" } ] }
6
MappingAgent — Маппинг на страницы сайта
Сопоставляет кластеры с существующими или новыми страницами, определяет действие
LLM Fallback Локальная обработка
Входные данные
  • clusters — список кластеров
  • existing_pages — URL из sitemap клиента
  • business_context — контекст бизнеса
LLM обработка

Метод LLMService.map_to_pages. Temperature: 0.3. В промпт передаются кластеры и список существующих страниц.

Возможные действия:

  • create — создать новую страницу (нет подходящей существующей)
  • update — обновить существующую страницу, добавив/расширив контент
  • merge — объединить с похожей страницей, избежать каннибализации
  • faq — добавить вопрос в раздел FAQ (слишком узкий/специфичный интент)
  • skip — пропустить (нерелевантно, низкий приоритет, дубль)

Правила в промпте: при наличии близкой по смыслу страницы — update/merge; при уникальном интенте — create; конкретный вопрос — faq; дубль — skip. URL должен быть SEO-оптимизированным (латиница, дефисы, без спецсимволов).

Формат ответа: [{"cluster_name": "...", "recommended_page": "/path", "action": "create", "reason": "..."}].

Fallback (локальная обработка)

Если LLM недоступен:

  • Генерация URL через _keyword_to_url — упрощённая транслитерация русских букв в латиницу с заменой спецсимволов на дефисы
  • Простое сопоставление: если в существующих страницах найдено совпадение по первым 2 словам main_keyword — действие update, иначе — create
Выходные данные
{ "mappings": [ { "cluster_name": "Покупка ноутбуков", "main_keyword": "купить ноутбук", "keywords": ["купить ноутбук", "ноутбук цена"], "intent": "commercial", "recommended_page": "/kupit-noutbuk", "action": "create", "reason": "Новый коммерческий интент, не покрыт существующими страницами" } ] }
7
PrioritizationAgent — Приоритизация кластеров
Оценивает кластеры по 7 факторам, рассчитывает взвешенный priority score и определяет уровень приоритета
LLM Локальная формула Fallback
Входные данные
  • mappings — список маппингов от MappingAgent
  • business_context — контекст бизнеса
LLM обработка

Метод LLMService.score_priorities. Temperature: 0.2. Все кластеры передаются одним запросом. LLM оценивает каждый кластер по шкале 0-100 по 7 факторам:

  • business_value — важность для бизнеса (доход, конверсия, LTV)
  • ranking_opportunity — реалистичность выхода в топ
  • intent_match — совпадение интента с предложением бизнеса
  • trend_growth — рост спроса (50 = стабильно)
  • content_gap — недостаток качественного контента у конкурентов
  • keyword_difficulty — сложность продвижения (0 = легко, 100 = очень сложно)
  • cannibalization_risk — риск каннибализации (0 = нет риска)

Формат ответа: [{"cluster_name": "...", "scores": {...}, "priority_score": 72.5, "priority_level": "high"}].

Локальная формула

Priority Score пересчитывается на сервере для надёжности:

priority_score = ( business_value * 0.25 + ranking_opportunity * 0.20 + intent_match * 0.20 + trend_growth * 0.10 + content_gap * 0.10 + (100 - keyword_difficulty) * 0.10 + (100 - cannibalization_risk) * 0.05 )

Уровни приоритета:

  • critical — priority_score ≥ 80
  • high — priority_score ≥ 60
  • medium — priority_score ≥ 40
  • low — priority_score < 40
Fallback (локальная обработка)

Если LLM недоступен:

  • Базовая оценка = min(размер кластера * 5, 50)
  • Бонус за действие: create +20, update +15, merge +10, faq +5
  • Ограничение 0-100, определение уровня по тем же порогам
  • Все оценки по факторам заполняются значением 50, cannibalization_risk = 0
Выходные данные
{ "results": [ { "cluster_name": "Покупка ноутбуков", "main_keyword": "купить ноутбук", "keywords": ["купить ноутбук", "ноутбук цена"], "intent": "commercial", "recommended_page": "/kupit-noutbuk", "action": "create", "priority_score": 78.5, "priority_level": "high", "reason": "...", "scores": { "business_value": 85, "ranking_opportunity": 70, "intent_match": 90, "trend_growth": 60, "content_gap": 75, "keyword_difficulty": 45, "cannibalization_risk": 10 } } ] }
8
FeedbackAgent — Обратная связь (заглушка)
Передаёт результаты дальше без изменений. В будущей версии — интеграция с Google Search Console
Локальная обработка
Входные данные
  • results — финальные результаты приоритизации
Обработка

Агент не изменяет данные. Добавляет метаинформацию:

  • feedback_status: "not_available"
  • message: "Мониторинг после публикации будет доступен в следующей версии"
Будущая функциональность

Планируется интеграция с Google Search Console для анализа позиций, CTR, сравнения план/факт.

Выходные данные
{ "results": [...], "feedback_status": "not_available", "message": "Мониторинг после публикации будет доступен в следующей версии" }
O
PipelineOrchestrator — Оркестратор
Управляет последовательным запуском агентов, SSE-прогрессом и сохранением результатов
Цепочка выполнения
agents = [ ("ResearchAgent", ResearchAgent()), ("ExpansionAgent", ExpansionAgent()), ("CleaningAgent", CleaningAgent()), ("IntentAgent", IntentAgent()), ("ClusteringAgent", ClusteringAgent()), ("MappingAgent", MappingAgent()), ("PrioritizationAgent", PrioritizationAgent()), ("FeedbackAgent", FeedbackAgent()), ]

Данные передаются через слияние словарей: current_data = {**current_data, **result}. Каждый следующий агент видит все предыдущие результаты.

SSE (Server-Sent Events)

Прогресс передаётся клиенту в реальном времени через asyncio.Queue:

  • agent_start — агент начал работу (с номером шага и общим количеством)
  • agent_complete — агент завершился успешно
  • error — ошибка в агенте (pipeline прерывается)
  • pipeline_complete — все агенты завершены, передаётся total_keywords
Сохранение в БД

По завершении пайплайна результаты сохраняются в таблицу KeywordResult:

  • Каждый ключевой запрос — отдельная строка
  • Поля: run_id, keyword, cluster_name, intent, confidence (из IntentAgent), recommended_page, action, priority_score, priority_level, reason
  • Confidence для каждого keyword извлекается из keywords_with_intents по точному совпадению (lowercase)
Управление статусом

Таблица PipelineRun обновляется на каждом шаге:

  • status: pending -> running -> completed/failed
  • current_agent — имя текущего агента
  • progress — процент выполнения (0-100)
  • started_at, completed_at — временные метки
  • error_message — текст ошибки при статусе failed

Модели OpenRouter
